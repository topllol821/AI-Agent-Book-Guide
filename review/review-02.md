# Review 02 — Content Review for Chapter 3 & Chapter 4

## 本次完成內容

- **Chapter 3 — 記憶與知識庫**：Memory（使用者記憶）、Knowledge Base（知識庫）、RAG（檢索增強生成）、Embedding 的基本概念
- **Chapter 4 — 工具**：Tool / Tool Calling / Function Calling 的差別、MCP 標準化協定、五類工具分類、Sidecar 安全機制

## 哪些地方比原書更容易理解？

1. **Memory vs Knowledge Base 的對比**：原書分散在兩個小節，本導讀直接以比較表呈現核心差異（對象、內容、誰填寫、更新頻率），讀者一眼就能掌握。

2. **RAG 的定位**：原書花了較多篇幅在 Embedding 技術細節。本導讀從「RAG 不是知識庫，是從知識庫找東西的技術」這個常見誤解切入，讓讀者先建立正確的分類觀念。

3. **Tool Calling 的五步驟流程**：原書用文字描述，本導讀用步驟列表呈現告知→選擇→執行→送回→回答的完整循環，並清楚區分 LLM 跟 Agent 框架的責任。

4. **MCP 的比喻**：用 USB-C 標準比喻 MCP，讓沒有技術背景的讀者也能理解標準化的重要性。

5. **各章開頭的「需要的問題」**：直接從讀者的日常體驗出發（「為什麼 ChatGPT 不記得我上次說了什麼？」），降低進入障礙。

## 哪些地方補充了最新 AI 發展？

1. **Anthropic Contextual Retrieval**（2025）：第三章加入業界補充，說明傳統 RAG 的改進方向。

2. **MCP 生態現況**（2026）：第四章加入業界補充，說明目前主流框架對 MCP 的支援狀況，並提醒 MCP 的安全挑戰。

3. **Sidecar 機制的實作說明**：原書有提及但較簡略，第四章用 Mermaid 圖示呈現 Sidecar 審查流程，並說明為何 Sidecar 刻意不看 LLM 的自由文字。

## 哪些內容刻意沒有深入？

1. **Embedding 的數學原理**：只保留「意思相近→距離相近」的直覺，沒有進入向量空間的數學細節。

2. **向量資料庫的比較**：沒有比較 Pinecone / Weaviate / Chroma / Qdrant 等產品的優劣——初學者不需要，需要時再選即可。

3. **稠密嵌入 vs 稀疏嵌入**：僅在「可跳過」區塊提及，沒有深入展開。

4. **MCP 傳輸層規格**：stdio 和 Streamable HTTP 的比較沒有寫入——方向正確，因為初學者不需要這個層級的細節。

5. **工具選擇的數學模型**：完全跳過，留到需要的人自行研究。
