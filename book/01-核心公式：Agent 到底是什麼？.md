# 01 — 核心公式：Agent 到底是什麼？

## 這章主要回答什麼問題？

2025 年開始，你突然發現 GitHub 上到處是 Agent、Cursor 幫你寫程式、Manus 幫你訂機票、Deep Research 幫你做報告、Claude Code 直接操作你的終端機。

這些東西跟 ChatGPT 有什麼不一樣？

為什麼不能繼續用 ChatGPT 就好？

## 英文名詞（附中文）

| 英文 | 中文 | 一句話解釋 |
|------|------|-----------|
| Agent | 智慧代理／代理 | 能自主完成任務的 AI 系統 |
| LLM (Large Language Model) | 大型語言模型 | Agent 的大腦，負責思考與推理 |
| Context | 上下文 | 送給 LLM 的所有資訊（提示詞＋歷史紀錄） |
| Tool / Tool Calling | 工具／工具呼叫 | Agent 呼叫外部功能（搜尋、寫檔、執行程式碼） |
| ReAct | 推理與行動循環 | Reasoning + Acting，Agent 反覆思考→行動→觀察 |
| Trajectory | 軌跡 | Agent 從開始到結束的完整執行記錄 |
| Harness | 馬具工程 | LLM 之外的所有工程設計（約束、驗證、糾錯） |
| Workflow | 工作流程 | 固定的步驟序列（A→B→C→D） |
| Planning | 規劃 | Agent 拆解任務、制定步驟的能力 |

## 作者真正想表達什麼？

**ChatGPT 是你問一句、它答一句。Agent 是你交代一件事、它自己做完。**

這是兩者最根本的差別。

你對 ChatGPT 說「幫我查一下最近一個月的比特幣走勢」，它會給你一段文字回答——然後沒了。

但如果一個 Agent 接到同樣的指令，它會自己去搜尋、下載歷史資料、寫 Python 做分析、畫圖表、最後給你一份完整報告。**整個過程不需要你不斷下指示。**

作者用一句話概括整本書的核心：

> **Agent = LLM + Context + Tools**

- **LLM（大腦）**：負責理解、推理、決策
- **Context（眼睛）**：決定 Agent 能看到什麼資訊
- **Tools（手腳）**：決定 Agent 能做什麼事

這三個缺一不可。沒有上下文的 LLM 等於蒙著眼睛；沒有工具的 LLM 只能紙上談兵；沒有 LLM 的上下文和工具只是死資料。

### 作者還有一個很重要的觀點

> **「實踐在前，命名在後」**

意思是你不需要先搞懂所有理論才能開始。先動手做，在做的過程中理解——這跟本書導讀的理念是一致的。

## 白話解釋

想像你請一個助理幫你訂機票：

```
你 → 助理 → 完成
```

**ChatGPT 模式：**

| 步驟 | 內容 |
|------|------|
| 你 | 「東京機票多少錢？」 |
| ChatGPT | 「大概一萬五左右。」 |
| 你 | （自己開瀏覽器、自己查航班、自己訂票） |

ChatGPT 只能回答，不能做事。它沒有手腳。

**Agent 模式：**

| 步驟 | 內容 |
|------|------|
| 你 | 「幫我訂下週去東京的機票，靠窗座位。」 |
| Agent 思考 | 需要查航班、比價、登入會員、選座位、付款 |
| Agent 行動 | 打開瀏覽器 → 搜尋航班 → 比對價格 |
| Agent 觀察 | 三個選項：A 航 $300、B 航 $350、C 航 $280 |
| Agent 再思考 | 選最便宜的 C 航，但轉機時間較長；使用者偏好靠窗 |
| Agent 再行動 | 登入會員 → 選 C 航靠窗座位 → 刷卡付款 |
| Agent 回覆 | 「已完成，訂單編號 XY123，C 航靠窗座位 $280。」 |

這個「思考→行動→觀察→再思考」的循環，就是 **ReAct**。

### 為什麼不乾脆把所有的步驟寫死就好？

這是初學者最常問的問題。

假設你寫一個訂機票的程式：

```
1. 去 A 網站查航班
2. 比對價格
3. 登入會員
4. 選座位
5. 刷卡
```

這個程式叫做「工作流程（Workflow）」。它最大的問題是：**遇到沒預料到的情況就掛了。**

- 如果 A 網站掛了？→ 程式掛掉
- 如果使用者說「我要靠走道」？→ 程式不知道怎麼辦
- 如果機票漲價了？→ 程式照刷，不會提醒你

Agent 不一樣。它會自己判斷：「A 網站掛了，我去 B 網站查」、「使用者說靠走道？好，我改選走道座位」、「機票漲了 30%？我會先問你要不要繼續」。

這就是 Agent 跟 Workflow 的本質差異：

| | Workflow | Agent |
|---|---|---|
| **步驟** | 固定 | 動態 |
| **彈性** | 零 | 高 |
| **例外處理** | 需要預先編寫 | 自主判斷 |
| **適合** | 穩定、可預測的任務 | 開放、變動的任務 |

## 真實案例

### 案例一：Cursor 寫程式（軟體開發）

你用 Cursor 說「幫我加一個深色模式切換按鈕」。它不會只給你一段程式碼貼上去——它會：

1. 讀取你的專案結構，理解用了什麼框架
2. 找出相關的 CSS 和元件檔案
3. 修改多個檔案
4. 執行 `npm run build` 確認沒錯誤
5. 告訴你改了哪些檔案、做了什麼

這就是一個 Coding Agent。

### 案例二：客服退款（企業）

一個客服 Agent 收到「我要退貨」的請求後：

1. 查詢訂單資料——這筆訂單還在嗎？
2. 確認退貨期限——超過 7 天了嗎？
3. 比對退款政策——這個商品可以退嗎？
4. 執行退款操作——如果是 VIP 客戶，直接退
5. 通知使用者——「已退款至原帳戶，預計 3-5 個工作天入帳」

全程不需要真人介入。但如果是同樣的 LLM 模型只是聊天（非 Agent），它只能回答：「好的，我理解您要退貨，已為您轉接客服。」

### 案例三：深度研究（AI）

你問：「東協十國首都之間，最近的一對距離多遠？」

普通 ChatGPT：會回答你可能知道的首都距離。

Agent（如 Deep Research）：會自己搜尋各國首都經緯度、寫 Python 計算所有配對的大圓距離、排序、找出最短的一對、給你答案和原始碼。你只問了一句話。

### 案例四：Claude Code 操作終端機（軟體開發）

你對 Claude Code 說：「幫我檢查這個專案有哪些套件可以升級。」

Claude Code 會：

1. 執行 `npm outdated` 查看過期套件
2. 逐一查看每個套件的 changelog，確認是否有 breaking changes
3. 更新 `package.json`
4. 執行 `npm install`
5. 跑測試確認一切都正常
6. 提交 commit

## 常見誤解

### ❌ 迷思一：Agent = ChatGPT 進階版

ChatGPT 是「對話工具」。Agent 是「任務執行系統」。

就好像計算機跟自動駕駛的差別——計算機幫你算答案，自動駕駛幫你開到目的地。

### ❌ 迷思二：Agent 會取代工程師

Agent 是工具，不是取代者。

就像 Excel 不會取代會計師——它改變了會計師的工作方式，但不會讓會計師消失。Agent 改變的是「你花多少時間在寫程式 vs 花多少時間在設計系統」。

### ❌ 迷思三：Agent = 工作流程（Workflow）

| | Workflow | Agent |
|---|---|---|
| 決定步驟的人 | 工程師 | Agent 自己 |
| 遇到意外 | 掛掉 | 自己想辦法 |
| 維護成本 | 每次改需求都要改程式 | 改提示詞或給新工具 |

### ❌ 迷思四：要把整個流程想清楚才能開始做

作者特別強調「實踐在前，命名在後」。不用等搞懂全部理論——先動手做一個簡單的 Agent，在做的過程中理解。

## 本章重點

1. **Agent 不是「更大的 ChatGPT」**——是「能自主完成任務的系統」

2. **核心公式：Agent = LLM + 上下文 + 工具**

   ```
   Agent = LLM + Context + Tools
           大腦    眼睛    手腳
   ```

3. **LLM 是大腦**：負責理解、推理、決策

4. **上下文是眼睛**：決定 Agent 能看到什麼資訊

5. **工具是手腳**：決定 Agent 能做什麼事

6. **ReAct 循環**：思考（Reasoning）→ 行動（Acting）→ 觀察（Observation）→ 再思考

7. **軌跡（Trajectory）**是 Agent 從開始到結束的完整執行記錄——每個思考、每個行動、每個觀察

8. **工作流程（Workflow）是固定的**，Agent 是動態的——兩者的差別比你想像的大

9. **Harness** 是模型之外的工程設計（約束、驗證、糾錯），往往比模型本身更重要

10. **實踐在前，命名在後**——不用等搞懂全部理論，先動手做
