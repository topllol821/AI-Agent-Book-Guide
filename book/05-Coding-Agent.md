# 05 — Coding Agent——最強的通用能力

## 這章主要回答什麼問題？

為什麼說「寫程式」是 Agent 最核心的能力？

以前讓 AI 寫程式有什麼問題？

Coding Agent 到底改變了什麼？

## 英文名詞（附中文）

| 英文 | 中文 | 一句話解釋 |
|------|------|-----------|
| Coding Agent | 程式碼代理 | 能自主理解、修改、測試程式碼的 Agent |
| Code Interpreter | 程式碼解釋器 | 在沙盒中執行 Python 程式碼的工具 |
| Sandbox | 沙盒 | 安全的隔離執行環境 |
| Linter | 程式碼靜態檢查工具 | 自動檢查程式碼風格和錯誤的工具 |
| Sessionless | 無會話設計 | 不需要安裝或登入，隨時發送訊息就得到回應 |
| ACI (Agent-Computer Interface) | Agent-電腦介面 | 讓 Agent 而非人類能順暢操作的工具設計原則 |

## 作者真正想表達什麼？

### 概念誕生故事：從 Code Generation 到 Coding Agent

要理解 Coding Agent，先看 AI 寫程式是怎麼演進的。

**第一代：程式碼片段生成（2021-2023）**

```
使用方式：
  開發者寫一段註解：「// 計算兩數的最大公因數」
  AI 回傳一段程式碼。

問題：
  ① AI 只看你給的那段註解——它不知道你的專案用什麼框架、什麼語言版本、什麼命名慣例
  ② 它只生成「片段」，你還是要自己決定貼到哪個檔案、哪個位置
  ③ 生成的程式碼沒有經過測試，你不知道對不對
  ④ 如果程式碼需要修改，你要重新描述一次

核心限制：
  AI 能寫程式，但不能理解你的專案。
```

**第二代：檔案內補齊（2023-2024）**

```
使用方式：
  開發者在 IDE 中打字，AI 根據游標位置補齊下一行程式碼。
  GitHub Copilot 是代表。

進步：
  ① AI 能看到游標所在的檔案內容
  ② 補齊速度很快，適合小幅編寫

還是沒解決：
  ① AI 只看一個檔案，不知道其他檔案的內容
  ② 它不會主動去修改多個相關檔案
  ③ 它不會幫你跑測試、不會 debug
```

**第三代：Coding Agent（2024-至今）**

```
使用方式：
  開發者說：「幫我把登入改造成 OAuth。」
  Agent 自己搜尋相關檔案、修改多個檔案、跑測試、提交。

改變：
  ① Agent 能搜尋整個專案（不只當前檔案）
  ② Agent 能同時修改多個檔案（不只補齊一行）
  ③ Agent 能跑測試確認沒問題（不只生成）
  ④ Agent 能自己 debug 錯誤（不像以前你看到錯誤要自己修）

所以 Coding Agent 不是「更強的程式碼生成」。
它是「能自主完成程式設計任務的系統」。
```

### 為什麼 AI 需要會寫程式？

程式碼生成不只是工具箱裡的一個工具——它是**元能力**。

意思是：它能在執行時期動態創造出新的工具和能力。

```
缺少某種格式的解析器？→ 寫一個。
需要計算某個數學公式？→ 寫 Python。
資料庫查詢太複雜？→ 寫 SQL。
圖表要特定樣式？→ 寫 matplotlib。
```

程式碼讓 Agent 從「工具使用者」變成「工具創造者」。

### 作者的核心論點

> **一個以開放任務為目標的通用 Agent，核心就是一個 Coding Agent 加上檔案系統。**

為什麼是檔案系統？因為檔案是 Agent 的中樞——記憶存成檔案、知識存成檔案、產物也存成檔案。檔案系統是 Agent 唯一不需要預先定義的儲存介面。

## 白話解釋

### 為什麼 Coding Agent 是目前最成熟的 Agent 類型？

你可能會想：跟寫程式比起來，訂機票、叫外送、查天氣不是對一般人更有用嗎？

為什麼偏偏是「寫程式」的 Agent 最成熟？

答案很簡單：**軟體工程有全世界最完整的自動化驗證基礎設施。**

| 領域 | 如何確認做對了？ |
|------|----------------|
| 寫程式 | 跑測試就知道了 |
| 訂機票 | 要真的去網站查才知道 |
| 叫外送 | 要看訂單狀態才知道 |
| 客服回覆 | 要客戶滿意才知道（而且不一定準） |

寫程式有測試、有 Linter、有型別檢查、有 CI/CD——這些都是自動化驗證工具。Agent 寫完程式，只要測試通過，就代表它做對了。這個「可驗證性」是其他領域不具備的。

### Coding Agent 的七個核心工具

一個基礎的 Coding Agent 只需要七個工具就能運作：

| 工具 | 用途 | 生活中的對應 |
|------|------|-------------|
| Code Interpreter | 在沙盒中執行 Python | 你寫完程式後「按執行」 |
| Bash Shell | 執行終端命令 | 你打開命令列下指令 |
| Read File | 讀取檔案內容 | 你打開檔案閱讀 |
| Write File | 寫入整個檔案 | 你從頭寫一個檔案 |
| Edit File | 修改現有檔案的部分內容 | 你修改檔案中的一段 |
| Glob (搜尋檔名) | 依檔名模式找檔案 | 你在檔案總管搜尋 |
| Grep (搜尋內容) | 依內容搜尋檔案 | 你在所有檔案中搜尋關鍵字 |

這七個工具，幾乎可以完成任何程式設計相關的任務。

## 真實案例

### 案例一：TODO 清單整理（生活）

你對 Agent 說：「幫我把專案裡所有 TODO 註解整理成一個清單。」

Coding Agent 會：

1. 用 Grep 搜尋所有檔案中的 `TODO` 關鍵字
2. 讀取有 TODO 的檔案，取得前後文
3. 用 Write File 寫入一個 `TODO.md` 檔案
4. 告訴你整理了幾個 TODO，在哪幾個檔案中

只用了兩個工具（Grep + Write）就完成了。流程中完全不需要 LLM 進行複雜推理——它就是按步驟執行。

### 案例二：大規模程式碼遷移（企業）

一家大型科技公司需要將數百萬行程式碼從一個 API 版本遷移到另一個。

關鍵不在模型多強——Claude、GPT-4、Gemini 都能正確改寫每一行程式碼。真正讓這個專案成功的，是 Harness 工程做對了三件事：

1. **知識存在於程式碼庫本身**：不需要外部文件，Agent 讀程式碼就知道要做什麼
2. **約束編進 Linter 和 CI**：改完的程式碼如果不符合規範，Linter 直接擋住
3. **驗證和糾正全鏈路自動化**：有測試可以跑、有 CI 可以驗證、有問題可以回退

### 案例三：記憶作為檔案（AI）

OpenClaw 把 Agent 的長期記憶存成一個 `MEMORY.md` 檔案。

使用者可以直接打開檔案閱讀和修改——如果 Agent 記錯了，直接刪掉那一行就行。這比向量資料庫更透明、更可控。

這體現了作者的核心觀點：**檔案系統是 Agent 的中樞。**

### 案例四：自動修復 CI 錯誤（軟體開發）

你 push 程式碼後 CI 失敗了。

你對 Agent 說：「幫我看為什麼 CI 失敗了。」

Agent 會：

1. 讀取 CI log（Read File）
2. 分析錯誤訊息（LLM 推理）
3. 找到出錯的程式碼（Grep + Read File）
4. 修改程式碼（Edit File）
5. 重新執行測試（Bash）
6. 如果還有錯誤，回到步驟 2

這就是 ReAct 循環在 Coding Agent 中的實際運作。

## 常見誤解

### ❌ 迷思一：Coding Agent = 給 ChatGPT 一個寫程式的提示

不對。Coding Agent 需要完整的工作流程：理解需求→搜尋相關程式碼→修改→測試→審查。遠比一個提示複雜。

### ❌ 迷思二：模型越強，Coding Agent 就越強

不對。LangChain 只透過優化 Harness（不改模型）就將 Terminal Bench 的成績從 52.8% 提升到 66.5%。Harness 的影響往往比模型大。

### ❌ 迷思三：Coding Agent 只對工程師有用

不對。如果你不會寫程式，Coding Agent 反而對你更有用——因為你無法自己寫程式，Agent 是你唯一的路。

### ❌ 迷思四：沙盒可以完全保證安全

不對。沙盒只是第一層防線。提示注入、資料外洩、忠誠度問題——這些需要多層防禦。

## 一句話記住

> **Coding Agent 不是更強的程式碼生成，而是能自主完成程式設計任務的系統。它的核心七工具讓它從工具使用者變成工具創造者。**

## 本章重點

1. **Coding Agent 的三代演進**：片段生成 → 行內補齊 → 自主完成任務

2. **Coding Agent 的核心七工具**：Code Interpreter、Bash、Read、Write、Edit、Glob、Grep

3. **程式碼是元能力**——能動態創造新工具的能力

4. **檔案系統是 Agent 的中樞**：記憶、知識、產物都存成檔案

5. **Coding Agent 最成熟的原因**：軟體工程有最完整的自動化驗證基礎設施（測試、Linter、CI）

6. **Harness 比模型更重要**：LangChain 不換模型就提升了 13.7%

7. **Coding Agent 對非工程師更有用**——因為你無法自己寫程式

8. **工作流程比模型重要**：搜尋→修改→測試→審查→交付

9. **安全永遠是前提**：沙盒隔離、權限控制、審查機制

10. **Sessionless 設計**：不需要安裝登入，隨時發送訊息就得到回應
