# 09 — 多模態與即時互動——跳出對話框

## 這章主要回答什麼問題？

為什麼文字不夠？

Agent 怎麼看懂圖片、聽懂語音、即時對話？

多模態和即時性的挑戰是什麼？

OpenAI、Anthropic、Google、Microsoft 各自在怎麼做？

## 英文名詞（附中文）

| 英文 | 中文 | 一句話解釋 |
|------|------|-----------|
| Multimodal | 多模態 | 同時處理文字、圖片、語音、影片等多種資訊形式 |
| ASR (Automatic Speech Recognition) | 語音辨識 | 把語音轉成文字 |
| TTS (Text-to-Speech) | 語音合成 | 把文字轉成語音 |
| VAD (Voice Activity Detection) | 語音活動檢測 | 判斷使用者有沒有在說話 |
| Full-Duplex | 全雙工 | 同時聽和說，像真正的對話 |
| Cascaded Architecture | 級聯架構 | 多個模型串接（語音→文字→LLM→語音） |
| Omni Model | 全端模型 | 一個模型同時處理多種模態 |
| Vision Language Model | 視覺語言模型 | 能看懂圖片的語言模型（如 GPT-4V、Claude Vision） |
| VLA (Vision-Language-Action) | 視覺-語言-動作模型 | 能看、能理解、能控制機器人的模型 |

## 作者真正想表達什麼？

### 概念誕生故事：為什麼文字不夠？

從第一章到第八章，我們討論的所有 Agent 能力都在文字世界裡。

但真實世界不是只有文字。

**第一階段：純文字 Agent**

```
使用者傳給 Agent 一張照片：
「這張桌子多少錢？」

Agent 回答：
「抱歉，我無法查看圖片。
請用文字描述這張桌子。」

使用者：
「⋯⋯」

Agent 的限制：
  ① 它看不到圖片、聽不到語音
  ② 它無法理解使用者的真實情境
  ③ 它在文字世界裡很強，但真實世界是視覺和聽覺的
```

**第二階段：分段處理（級聯架構）**

```
語音輸入：
  VAD（判斷說完了）→ ASR（語音轉文字）→ LLM（理解）→ TTS（文字轉語音）

圖片輸入：
  OCR（文字辨識）→ 把文字塞進 Context → LLM 回答

進步：
  ① 能處理語音和圖片了
  ② 技術成熟，容易實作

問題：
  ① VAD 要等使用者說完才開始處理——延遲 500-800ms
  ② 語調、情緒、停頓——這些資訊在轉成文字時就丟失了
  ③ OCR 只能認文字，不能理解圖片的視覺內容
```

**第三階段：原生多模態**

```
一個模型直接看圖片、聽語音、讀文字。

進步：
  ① 不需要先 OCR 再 LLM——模型直接看圖片理解內容
  ② 語調和情緒得以保留
  ③ 串流處理，不用等整段說完

新的限制：
  ① 計算成本高——處理圖片和語音比處理文字貴很多
  ② 即時性要求高——語音對話需要 200ms 內回應
  ③ 模型變大、變慢、變貴
```

### 即時性讓多模態問題變得更難

這是全書最關鍵的洞察之一。

靜態看一張圖片，模型可以想多久就想多久。但語音對話不行——你不可能讓人類等 3 秒才得到回應。

即時性把問題從「能不能做」變成了「能不能在時間預算內做完」。

```
靜態任務（不限時間）：
  看一張發票照片，提取金額和日期
  → 模型可以花 5 秒、10 秒、甚至 30 秒
  → 使用者不介意

即時任務（限制時間）：
  使用者說「幫我查一下明天的天氣」
  → 模型需要在 200-500ms 內開始回應
  → 超過 1 秒，使用者就覺得「卡卡的」
```

## 白話解釋

### 三種語音架構的演進

目前主流的語音 AI 架構有三代：

| 架構 | 做法 | 延遲 | 優點 | 缺點 |
|------|------|------|------|------|
| 級聯 | 接力賽：VAD→ASR→LLM→TTS | 0.9-2 秒 | 技術成熟、易除錯 | 延遲高、語調遺失 |
| 全端（Omni） | 一個模型直接處理音訊 | 0.3-0.8 秒 | 延遲低、保留語調 | 仍假設輪流說話 |
| 全雙工 | 邊聽邊說，可打斷、可插話 | < 0.2 秒 | 像真人對話 | 技術難度高、成本高 |

```
生活比喻：

級聯：像你跟外國人對話，你說一句，他翻譯給自己聽，然後回答，再翻譯回來。
       每一棒都要等上一棒。

全端：像你跟一個懂中文的外國人對話。他直接聽懂你的中文，用中文回答。
       還是輪流說，但他反應快多了。

全雙工：像你跟朋友聊天。
        你們可以同時說、可以打斷、可以用「嗯嗯」表示正在聽。
        這是最自然的對話方式。
```

### 主要廠商的策略方向

目前各大 AI 公司在多模態上的策略不同：

| 公司 | 方向 | 代表產品 | 特點 |
|------|------|---------|------|
| OpenAI | 全端語音 + Vision | GPT-4o、GPT-4.1 | 從文字出發，逐步加入語音和視覺 |
| Anthropic | Vision 優先 | Claude 3.5 Sonnet、Claude Opus | 擅長圖表和文件理解，語音較晚推出 |
| Google | 原生多模態 | Gemini 2.0 Pro | 從訓練開始就是多模態，原生整合 |
| Microsoft | 平台化整合 | Azure AI Vision、Copilot | 強調企業場景的工具鏈整合 |

> **【業界補充】** 各家策略的選擇反映了一個根本取捨：**全端模型 vs 最佳組件組合。** OpenAI 和 Google 傾向訓練一個超大模型處理所有模態；Anthropic 傾向讓專業模型做專業事，再用工程整合。截至 2026 年中，兩種做法各有優劣——全端延遲低但成本高，組合式靈活但工程複雜。

### 文字 vs Vision vs Multimodal 比較

| 能力 | Text-only | + Vision | + Multimodal (全模態) |
|------|-----------|---------|----------------------|
| 讀文字 | ✅ | ✅ | ✅ |
| 看圖片 | ❌ | ✅ | ✅ |
| 聽語音 | ❌ | ❌ | ✅ |
| 即時對話 | ❌ | ❌ | ✅ |
| 理解圖表 | ❌ | ✅ | ✅ |
| 看懂影片 | ❌ | 逐幀可看 | ✅ 串流 |
| 成本 | 低 | 中 | 高 |

## 真實案例

### 案例一：拍照問問題（生活）

你在 IKEA 看到一張桌子，想知道它能不能自己組裝。

你拍照傳給 Agent：

- **文字 Agent**：「請描述這張桌子的外觀。」
- **Vision Agent**：（看到照片）「這是 IKEA 的 LAGKAPTEN 桌面，型號 104.763.45，需要自行組裝。說明書在這裡。」
- **Multimodal Agent**：同上，但你可以接著用語音問「那桌腳要去哪裡買？」——不用打字。

### 案例二：即時語音客服（企業）

你打電話給銀行客服，接電話的是 AI。

- **級聯架構**：「請您⋯⋯說出⋯⋯您的⋯⋯會員⋯⋯編號。」（每句之間停 1 秒）
- **全雙工架構**：你說「我要查一下我的信用卡帳單。」Agent 在你停頓時立刻回應「好的，請說出生日驗證。」同時聽到你含糊說了「一月⋯⋯」，Agent 可以打斷確認「是一月十五日嗎？」

全雙工的體驗接近真人客服。級聯的體驗像跟機器講話。

### 案例三：電腦操作 Agent（AI）

Agent 需要操作瀏覽器填寫表單。

- **沒有 Vision**：只能透過 DOM 或 API 操作——如果頁面是圖片渲染的，完全無法處理。
- **有 Vision**：每次截圖 → Vision LLM 分析畫面 → 決定下一步 → 操作。可以處理任何網頁，不管它是怎麼渲染的。

Google 的 Project Mariner 和 Anthropic 的 Computer Use 都是這個方向。

### 案例四：倉儲機器人（工廠）

機器人需要從貨架上拿取特定商品。

```
純文字的 Agent：
  知道「拿藍色盒子」，但不知道哪個盒子是藍色。
  → 無法執行。

有 Vision 的 Agent：
  看到貨架上有多個盒子。
  識別出「藍色、方形、標籤朝外的那個」。
  → 可以執行。

有 VLA 的 Agent：
  看到 → 理解 → 規劃抓取姿勢 → 控制機械臂。
  整個過程在數十毫秒內完成。
  → 可以自主完成。
```

## 常見誤解

### ❌ 迷思一：多模態 = 同時處理所有模態

不對。靜態的圖片理解（看一張截圖）相對成熟。真正的挑戰是「即時的多模態」——語音串流、持續變化的螢幕畫面、機器人的即時感測。

### ❌ 迷思二：VAD 只是小問題

不對。VAD 的 500-800ms 等待是語音互動中最難消除的延遲。全雙工架構之所以重要，就是因為它取消了「等使用者說完」的步驟。

### ❌ 迷思三：全端模型一定比級聯好

不一定。全端模型延遲更低、保留更多資訊，但成本更高、更難除錯。在一些場景中，級聯仍然是更好的工程選擇。

### ❌ 迷思四：Computer Use = 截圖 + LLM

截圖只是第一步。真正的 Computer Use 需要：連續畫面理解、座標定位、滾動和點擊的精確控制、表單填寫的欄位辨識。每個環節都有 engineering 挑戰。

## 本章重點

1. **前三代的文字 Agent 無法處理真實世界**——真實世界有圖像、語音、影片

2. **三種語音架構**：
   - 級聯（VAD→ASR→LLM→TTS，0.9-2s）
   - 全端（一個模型處理全部，0.3-0.8s）
   - 全雙工（邊聽邊說，< 0.2s）

3. **即時性把問題變難**——靜態看圖不限時間，語音對話必須在 200ms 內回應

4. **OCR 是 Vision 1.0**——只能認文字，不能理解畫面內容

5. **Vision 2.0**——模型直接看懂圖片（GPT-4V、Claude Vision、Gemini）

6. **各家廠商策略不同**：OpenAI 全端、Google 原生多模態、Anthropic Vision 優先、Microsoft 平台整合

7. **Computer Use**：Agent 像人一樣操作電腦螢幕

8. **VLA（視覺-語言-動作）**：AI 理解畫面後直接控制機器人

9. **全雙工是語音互動的終極目標**——但技術和成本挑戰很大

10. **選擇哪種架構取決於場景**：不是最新的就是最好的
