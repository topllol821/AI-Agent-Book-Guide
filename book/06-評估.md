# 06 — 評估——沒有度量就沒有改進

## 這章主要回答什麼問題？

為什麼 Prompt 在測試的時候看起來很好，上線卻一直失敗？

你怎麼知道你換了一個模型後，效果是變好還是變差？

為什麼同一個 Agent，有時候表現很好、有時候完全不行？

## 英文名詞（附中文）

| 英文 | 中文 | 一句話解釋 |
|------|------|-----------|
| Evaluation | 評估 | 衡量 Agent 系統表現的方法與流程 |
| Benchmark | 基準測試 | 公開的標準化考試，用來比較不同模型 |
| Rubric | 評分標準 | 評估的評分表，定義什麼是「好」的答案 |
| LLM-as-a-Judge | 用 LLM 當評審 | 讓另一個 LLM 幫你打分數 |
| Pass@k | k 次中至少一次成功 | 測量 Agent 的上限能力 |
| Pass^k | k 次全部成功 | 測量 Agent 的穩定度 |
| Ablation Study | 消融實驗 | 關掉某個組件，看整體表現的變化 |
| Regression | 退化 | 新版本不如舊版本 |

## 作者真正想表達什麼？

### 概念誕生故事：為什麼需要評估？

這是一個很多團隊實際經歷過的故事。

**第一階段：Prompt 看起來很好**

```
你寫了一個很漂亮的 Prompt。
你測了 3 個案例——都過關了。
你覺得：「太好了，可以上線了。」

上線第一天：
  第一個使用者——沒問題。
  第二個使用者——看起來沒問題。
  第十個使用者——完全答錯。
  你回去重讀 Prompt——看不出來哪裡有問題。

為什麼？
  ① 你只測了 3 個案例，但實際上有 300 種變化
  ② 你測的都是你想到的案例——真正的錯誤往往發生在你沒想到的地方
  ③ Prompt 不是你寫錯，而是某些邊界情況 LLM 就是應付不來
```

**第二階段：人工測試**

```
你找了 5 個人來測試。
每一個案例人工審查對不對。

問題：
  ① 10 個案例：5 分鐘看完
  ② 100 個案例：需要專人盯 1 小時
  ③ 1000 個案例：不可能

而你的 Agent 每天處理 10 萬次請求——人工測試的覆蓋率根本跟不上。
```

**第三階段：自動化評估**

```
你意識到：
  ① 不能只靠人工——太慢、成本太高
  ② 不能只測 3 個案例——覆蓋率不夠
  ③ 不能靠感覺——你需要數據

所以評估系統出現了：
  ① 建立一組測試用例（像考試題庫）
  ② 定義評分標準（像考試答案）
  ③ 自動化執行和評分（像電腦閱卷）
  ④ 持續追蹤分數變化（像成績單）
```

### 作者真正想解決的問題

> **如果你沒有評估機制，你根本不知道你的 Agent 是真的變好了，還是只是運氣好。**

這是作者反覆強調的核心觀點。

他認為評估的第一價值不是「給當前系統打分」，而是**讓你能快速、可靠地跟上模型的演進**。

```
情境：
  今天 OpenAI 發佈了新模型，號稱更強、更便宜。
  你的競爭對手打算下週切換過去。

沒有評估系統：
  你只能靠直覺：「嗯⋯⋯我覺得新模型好像比較好？」
  或者等社群反饋——至少等一週。

有評估系統：
  你在 2 小時內跑完 500 個測試用例。
  數據顯示：新模型在你的任務上準確率提升 12%，成本降低 40%。
  你當天就完成切換。
```

### 為什麼 Prompt 看起來很好，上線卻失敗？

這是最常見的痛點。**你的 Demo 成功了，不代表你的系統準備好了。**

原因有三：

```
原因一：測試案例太少
  你測了 3 個案例，但 Agent 面對的是 300 種情境。
  你的 3 個案例恰好都是「常見的、簡單的」情況。
  真正的問題發生在「少見的、複雜的」情況。
  Demo 不會測到這些——因為你在展示時會刻意挑選「一定會成功」的例子。

原因二：你測試的是「輸入」，不是「輸出」
  你測試時看的都是「我問這個問題，它答得對不對」。
  但上線後真正的問題是：
  - 使用者輸入惡意內容（提示注入）→ 你沒測過
  - 使用者問了模糊的問題 → 你沒測過
  - 工具回傳了預期外的結果 → 你沒測過
  Demo 的環境是受控的，上線的環境不是。

原因三：LLM 的回覆有隨機性
  同一個 Prompt、同一個問題，兩次回答可能不同。
  你測試時剛好遇到對的——上線時它給出錯的了。
  Demo 成功一次就夠了，但生產環境需要每一次都成功。
```

**所以評估存在的真正目的，不是測試，而是降低上線風險。**

測試是「確定它會動」。評估是「確定它在各種情況下都不會出大問題」。

| | 測試（Testing） | 評估（Evaluation） |
|---|---|---|
| **目的** | 確認功能正常 | 量化風險與表現 |
| **範圍** | 你想到的案例 | 你沒想到的邊界 |
| **頻率** | 開發階段做幾次 | 持續做、每次變更都做 |
| **失敗的後果** | 改 code | 上線出包、使用者崩潰 |

如果你只做測試不做評估，你的系統「應該能用」。如果你做了評估，你才知道**它什麼時候不能用**。

## 白話解釋

### 評估不是考試，是一種「決策工具」

你可能會覺得：評估就是考 Agent 嘛，看它能得幾分。

但評估的真正用途是：**幫你做決策。**

```
你面對的問題         → 評估能幫你回答
────────────────────────────────────
該不該換模型？       → 跑一次評估，看分數變化
Prompt 這樣改好嗎？   → 跑一次評估，比較前後分數
這個工具有用嗎？     → 做消融實驗，關掉它看分數變化
系統有退化嗎？       → 對比上週的分數
```

### 兩種不同的成功標準

評估時最常搞混的是這兩種指標：

| | Pass@k | Pass^k |
|---|---|---|
| **意思** | k 次中至少成功 1 次 | k 次全部成功 |
| **測什麼** | 能力上限 | 穩定度 |
| **數字範例** | 如果單次成功率 60% | |
| | Pass@5 ≈ 99% | Pass^5 ≈ 7.8% |
| **什麼時候用** | 你可以在多個結果中挑一個好的 | 每次結果都必須正確（如退款的 Agent） |

差別非常大：60% 的單次成功率下，5 次中至少 1 次成功的機率是 99%（幾乎保證有），但 5 次全部成功的機率只有 7.8%（非常難）。

### 怎麼設計一個評估？

作者建議的 Rubric（評分標準）四個準則：

```
準則一：基於專家指導
  評分標準應該反映領域知識，而不是通用的「答得好不好」。
  例如客服評估：政策合規性比回答長度重要得多。

準則二：全面覆蓋
  至少要包含：
  - 事實準確性（數字、日期對不對）
  - 邏輯連貫性（推理有沒有漏洞）
  - 資訊完整性（有沒有遺漏關鍵資訊）
  - 安全性（有沒有洩漏機密）

準則三：權重設計
  不是每個項目同等重要。
  - 必要項：錯了就當作失敗（如幻覺、洩漏個資）
  - 重要項：錯了扣分但還可接受
  - 可選項：錦上添花

準則四：自包含
  每個評分項目要讓評審者不需要額外領域知識就能判斷。
  不要寫：「請判斷這個答案是否符合公司政策。」
  要寫：「答案是 7 天。標準答案是 7-15 天。答 7 天算對。」
```

## 真實案例

### 案例一：個人寫作助手（生活）

你寫了一個 Agent 幫你改英文作文。

你用 3 個例句測試——都改得很好。你覺得可以用了。

上線後發現：
- 遇到專業術語（如醫學名詞）時，Agent 會亂改成一般用語（你沒測到）
- 遇到長句子（超過 30 字）時，Agent 會斷錯句（你沒測到）
- 遇到引號內的對話時，Agent 會錯誤修改（你沒測到）

正確的做法：建立 50 題測試集，包含專業術語、長句、對話、書信等不同類型，每次修改 Prompt 或換模型後全部跑一次。

### 案例二：客服退款的評估 Rubric（企業）

一個客服退款 Agent 的評估標準：

| 評分項目 | 權重 | 一票否決？ | 說明 |
|---------|------|-----------|------|
| 操作正確性 | 4 分 | 否 | 退款的步驟是否正確 |
| 政策合規性 | 4 分 | 是 | 違反退款政策直接不及格 |
| 資訊完整性 | 3 分 | 否 | 是否告知了退款時程 |
| 幻覺檢測 | — | 是 | 只要產生幻覺就失敗 |
| 安全性 | — | 是 | 洩漏個資或金鑰直接失敗 |

好的評估不只要測成功場景，還要測邊界——使用者要退 15 天前的訂單時，Agent 能否正確拒絕。

### 案例三：LLM-as-a-Judge（AI）

假設你要評估 1000 個客服回覆。

不可能全部人工看——太慢也太貴。

解法：讓另一個 LLM（如 GPT-4）當評審。

```
做法：
  1. 對每個客服回覆，給評審 LLM 一個 Rubric
  2. 評審 LLM 根據 Rubric 逐項評分
  3. 統計所有分數

要注意的偏見：
  ① 長度偏見：評審 LLM 傾向給更長的回覆更高分
  ② 自肥偏見：GPT-4 評審會偏好 GPT-4 產生的回答
  ③ 順序偏見：兩個回答放一起評時，先出現的比較有利

解法：
  ① 至少混合使用不同評審模型（如 GPT-4 + Claude）
  ② 在 Rubric 中明確禁止以長度作為評分標準
  ③ 做 A/B 比較時隨機交換順序
```

### 案例四：換模型的決策（企業）

你的客服 Agent 目前用 GPT-4o。OpenAI 發佈了 GPT-4.1，號稱更便宜、更快。

要不要換？

```
沒有評估系統：
  你讓 3 個人測試了 10 個案例——「感覺差不多」。
  你覺得：「省錢嘛，換吧。」
  上線後發現：GPT-4.1 在某些退款情境下會直接拒絕客戶——你沒測到。

有評估系統：
  你跑了一次完整的 500 題評估。
  數據顯示：
  - 一般客服回覆：分數差不多（GPT-4.1 還略高 2%）
  - 退款處理：GPT-4.1 在「超過 7 天」的案例中拒絕率 40%（GPT-4o 只有 5%）
  你決定：一般客服先換，退款處理等新版本。

  這就是評估的真正價值：不只是打一個分數，而是讓你看清楚每個環節的表現。
```

## 常見誤解

### ❌ 迷思一：評估 = 看排行榜

不對。排行榜測的是通用能力。你的任務需要特定能力。新模型在公開 Benchmark 上表現更好，不代表在你的特定任務上也更好。

### ❌ 迷思二：Pass@k 跟 Pass^k 是一樣的

完全不一樣。Pass@5 是「5 次中至少 1 次對」，Pass^5 是「5 次全部對」。兩者差距極大。

### ❌ 迷思三：讓模型自己檢查自己就好

不對。除非引入新資訊（如執行測試、渲染截圖），否則讓模型自己審查自己的輸出，效果通常為負。

### ❌ 迷思四：評估一次就夠了

不對。評估需要持續進行。換了模型、改了 Prompt、加了工具——每一次變更都應該重新評估。否則你無法知道改變的效果。

## 本章重點

1. **Prompt 測試過關 ≠ 上線沒問題**——因為你只測了你想到的案例

2. **評估的真正目的不是打分，是輔助決策**——該不該換模型、改 Prompt、加工具

3. **Pass@k 測上限，Pass^k 測穩定度**——兩者完全不一樣

4. **Rubric 四準則**：專家指導、全面覆蓋、權重設計、自包含

5. **LLM-as-a-Judge 能節省大量成本**，但要注意長度偏見、自肥偏見、順序偏見

6. **消融實驗**：關掉某個組件，看整體表現的變化——這是驗證組件價值的方法

7. **評估的對象不應只是模型**，而是模型與 Harness 的組合

8. **自我審查（無外部反饋）通常無效**——沒有新資訊的審查不會進步

9. **評估需要持續進行**，每次變更都應該重新跑一次

10. **完善的評估體系讓你能在 2 小時內判斷「換模型值不值得」**
